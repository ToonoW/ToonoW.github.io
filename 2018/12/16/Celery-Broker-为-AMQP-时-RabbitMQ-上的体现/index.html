Celery Broker 为 AMQP 时 RabbitMQ 上的体现详细

Python Celery
 2018/12/16

当 Celery 使用 RabbitMQ 作为 Broker 的时候至少可以提供以下特性：

  1. 自定义任务队列，worker 订阅指定的任务队列进行消费；
  2. 每一个 worker 订阅所有 worker 的心跳，和每一个 worker 的上下线通知；
  3. 对每一条任务的处理状态进行通知；
  4. 能够对一个或多个 worker 进行开关、worker 所在宿主机状态监控、调整进程数、设置协程数范围、指定队列实时流控、指定队列超时设置；
  5. 对 worker 的控制结果进行获取。

Celery 在使用 RabbitMQ 作为 Broker 的时候，使用了其 exchange queue 的功能，对以上提及的特性进行了实现，具体在 RabbitMQ 上的体现如下表：

Exchange Name Exchange Type Queue Name Routing Key Subscriber Mean
celery Direct celery celery all workers 默认任务队列
自定义 Direct 自定义 自定义 all workers 指定工作队列
celery.pidbox Fanout celery@{worker_name}.celery.pidbox per worker 对 worker 的控制指令
celeryev Topic celeryev.{uuid} worker.# per worker worker 状态信息，包括所有 worker 心跳和上下线
celeryev Topic celeryev.{uuid} # monitor worker 状态信息和任务处理状态信息
reply.celery.pidbox Direct 随机生成 随机生成 monitor 对 worker 的控制结果信息
  1. Subscriber 为 all worker 代表所有订阅此队列的 worker 都会消费此队列的消息，并且不同 worker 不会重复消费同一条消息；
  2. Subscriber 为 per worker 代表每个 worker 都会有对应一条队列进行订阅消费；
  3. Subscriber 为 monitor 代表此队列只有监控程序进行订阅和消费，监控程序例如有 flower；
  4. reply.celery.pidbox 此 exchange 产生的队列的名称和 routing key 都是由监控程序随机生成的，worker 作为此类队列的生产者，将控制结果发送到队列上，由监控程序进行订阅消费，消费结束后队列自动销毁；

    worker 的状态信息和任务状态信息的枚举和含义可参照此目录
